# AWS-Cloud-Resume

Emre Çancıoğlu's personal site: a resume/portfolio homepage plus an admin panel for manually tracking a TEFAS (Turkish mutual fund) portfolio and for uploading/analyzing credit card statements (AI-extracted spending by category). Hosted on AWS, deployed from this repo.

## Structure

```
apps/web/    Vite + React 19 + TypeScript + Tailwind v4 — the public site + admin panel UI
apps/api/    Lambda backend (Node 20, TypeScript, esbuild bundle) — REST-ish API behind API Gateway
infra/       Terraform — Cognito, DynamoDB, Lambda, API Gateway
```

No root-level build tooling — each app manages its own `package.json`/`node_modules`.

## apps/web (public site + admin panel)

- Public resume site: `src/sections/*`, `src/pages/Home.tsx`, English/Turkish i18n (`src/i18n/`), dark/light theme (`src/theme/`), all data hardcoded in `src/data/content.{en,tr}.ts`.
- Admin panel: lazy-loaded at `/admin/*` (see `src/pages/admin/AdminApp.tsx`) so the Cognito SDK + QR code lib never load for public visitors — only the public bundle (~435KB) loads on `/`; the admin chunk (~138KB) loads separately. Auth: `src/auth/AuthContext.tsx` wraps `amazon-cognito-identity-js` (SRP login, forced first-login password change, TOTP MFA setup via QR + verification, session persistence, silent token refresh via `getIdToken()`). `src/auth/cognitoConfig.ts` and `API_BASE_URL` are **hardcoded constants**, not env vars — same convention as the pre-existing visitor-counter Lambda URL (`src/lib/visitorCounter.ts`). These aren't secrets (public SPA client ids), so this is intentional, not a shortcut.
- **Always call the API through `getIdToken()`, never the static `idToken` field**, for anything other than a render-time display/gate check. `idToken` is a snapshot that goes stale after Cognito's 1-hour token validity window; `getIdToken()` (in `AuthContext`) transparently refreshes via the stored refresh token. `apiFetch()` (`src/auth/api.ts`) takes `getIdToken` as its first argument for this reason.
- Admin dashboard/fund management (`/admin`): `src/pages/admin/AdminDashboard.tsx` (add-fund form + total portfolio value card, fed by `GET /portfolio/summary`) and `FundRow.tsx` (per-fund edit/delete, price history CRUD, transaction/BUY-SELL CRUD, all inside an expandable row).
- Credit card statement analysis (`/admin/statements`, Faz 4): `src/pages/admin/Statements.tsx` — upload a PDF statement (gets a presigned S3 PUT URL from `POST /statements`, then PUTs the file directly to S3, browser→S3, bypassing the API Lambda entirely), polls `GET /statements` every 4s while any statement is `pending`/`processing`, shows a category-breakdown spending summary from `GET /spending/summary` (simple CSS bars, no chart library). Extraction/categorization happens server-side (see apps/api below) — this page only uploads and displays.
- Shared admin UI bits (`src/pages/admin/`): `formFields.tsx` (`IconField`, `SubmitButton`, `plainInputClass`, `iconButtonClass`), `Toast.tsx` (`useToast().showToast(message, "success"|"error")`, mounted once in `AdminApp.tsx`), `format.ts` (`currencyFormatter`/`numberFormatter`, `tr-TR`/TRY).
- `vite.config.ts` has `define: { global: 'globalThis' }` — required because `amazon-cognito-identity-js` references Node's `global`, which doesn't exist in browsers. Don't remove this.
- Commands: `npm run dev`, `npm run build` (`tsc -b && vite build`), `npm run test` (Vitest), `npm run lint` (oxlint).
- Shared focus-ring styles live in `src/styles/focusRing.ts` (`focusRing`, `focusRingInset`, `focusRingOnAccent`) — use these instead of writing `focus-visible:outline-*` classes inline.
- Test coverage exists for `AuthContext` (mocks `amazon-cognito-identity-js` via `vi.hoisted` + regular `function` mocks — arrow functions can't be used as constructors, `new (() => {})()` throws), `ProtectedRoute`, `AdminLogin`, `AdminDashboard`, `FundRow`, `Statements` (the latter three mock `../../auth/api`'s `apiFetch` directly).
- Testing gotcha: don't `screen.findByText("Akbank")` (or any bank name) on the Statements page without disambiguating — the bank `<select>`'s own `<option>` list contains the same names as real statement rows, causing "found multiple elements". Assert on something unique to the row instead (status text, etc). Also: native `required` on `<input type="file">` is unreliable in jsdom (silently blocks form submission with no error) — validate manually in the submit handler instead.

## apps/api (backend — two Lambda functions from one codebase)

- **API Lambda** (`src/handler.ts`, entry `dist/api/index.mjs`), routed by `event.routeKey` (API Gateway HTTP API v2 payload format). Routes (all except `/health` require a Cognito JWT, `Authorization: Bearer <idToken>`):
  - `GET /health` — no auth.
  - `GET /portfolio` — list funds (raw). `GET /portfolio/summary` — funds + `netUnits`/`latestPrice`/`currentValue` per fund (nets BUY-minus-SELL transaction units × latest price) + `totalValue`. Prefer `/summary` for anything UI-facing.
  - `POST /funds`, `PUT /funds/{fundCode}`, `DELETE /funds/{fundCode}` (cascade-deletes all price/transaction rows under that fund via `BatchWriteItem`, chunked by 25).
  - `POST|GET /funds/{fundCode}/prices`, `PUT|DELETE /funds/{fundCode}/prices/{date}`.
  - `POST|GET /funds/{fundCode}/transactions`, `DELETE /funds/{fundCode}/transactions/{txnId}` (txnId is a `randomUUID()`; delete looks it up via a Query + in-memory `find`, no GSI for it — fine at personal scale).
  - `POST /statements` — creates the `METADATA` row and returns a presigned S3 PUT URL (5 min expiry) for the browser to upload directly to. `GET /statements` (list, newest first), `GET /statements/{statementId}` (metadata + its transactions), `DELETE /statements/{statementId}` (cascade, same `BatchWriteItem` pattern as funds).
  - `GET /spending/summary?month=YYYY-MM` (month optional) — category totals + recurring-payment list across all credit card statements, same in-memory-reduce aggregation style as `/portfolio/summary`.
- **Processor Lambda** (`src/handler-processor.ts` → `processor.ts`, entry `dist/processor/index.mjs`) — S3-triggered (`ObjectCreated` on the statements bucket, `statements/` prefix, `.pdf` suffix), NOT reachable via API Gateway. Fetches the PDF from S3, reads the OpenAI API key from SSM Parameter Store (`getOpenAiKey()`), calls OpenAI's chat completions endpoint with the PDF as a base64 file input + a `response_format: json_schema` (Structured Outputs) request for `{transactions: [{date, merchant, amount, category, isRecurring}]}`, writes each as a `TXN#` row, flips the statement's `status` through `pending → processing → done` (or `failed` + `errorMessage`, then rethrows so the Lambda invocation shows as failed in CloudWatch too). Timeout 90s (external API call + multi-page PDFs). **The exact OpenAI request shape (`extractTransactions` in `processor.ts`) may drift from current API docs — verify before relying on it**, it was written from documentation research, not a live test call.
- **No Lambda environment variables anywhere in this codebase** — see "AWS account quirk" below. DynamoDB table name is a hardcoded constant (`src/dynamo.ts`); the OpenAI key is read from SSM Parameter Store **at invocation time**, not passed as an env var (this also sidesteps ever putting a real secret in Lambda console/CLI output).
- DynamoDB single-table design (see `infra/dynamodb.tf` header comment for the exact key schema): `pk=FUND#<code>` (`METADATA`/`PRICE#<date>`/`TXN#<date>#<txnId>`), `pk=STATEMENT#<id>` (`METADATA`/`TXN#<date>#<txnId>`), GSI1 `gsi1pk=FUND|PRICE|TXN|STATEMENT|CCTXN`/`gsi1sk=<sortKey>`.
- Shared response helpers in `src/http.ts` (`jsonResponse`, `parseBody`); shared S3 client/bucket constant in `src/s3.ts`.
- Test coverage: Vitest (`npm run test`), one file per route module, mocking `ddb.send`/`s3.send` — cast as `ddb.send as unknown as DdbSendMock` (from `src/test-utils.ts`), **don't use `vi.mocked(ddb.send)`** — the real `DynamoDBDocumentClient#send`/`S3Client#send` overload unions make TS infer an unusable type for mock setup. `handler.test.ts` mocks each route module's exports directly to test only the routing switch. `processor.test.ts` mocks `global.fetch` for the OpenAI call and `@aws-sdk/client-ssm`'s `SSMClient` (via a regular `function` mock returning `{send: mockSsmSend}` — same constructor-can't-be-arrow-function rule as the frontend's Cognito mocks).
- `npm run build` now produces **two** bundles: `npm run build:api` → `dist/api/index.mjs`, `npm run build:processor` → `dist/processor/index.mjs`. After changing anything under `apps/api/src`, run `npm run build` **and** `terraform apply` from `infra/` (each Lambda's `archive_file` zips its own `dist/api` or `dist/processor` folder — not deployed via GitHub Actions). `.github/workflows/ci-api.yml` runs typecheck/test/build on PRs and pushes touching `apps/api/**`, but never deploys.

## infra (Terraform)

- Resources: Cognito User Pool + admin user + app client (`cognito.tf`), DynamoDB table (`dynamodb.tf`), Lambda + API Gateway HTTP API + Cognito JWT authorizer + IAM roles + all routes (`lambda_api.tf`), AWS Budgets monthly cost alarm (`budget.tf`), statements S3 bucket + processor Lambda + its IAM role + S3 event notification (`statements.tf`, Faz 4).
- The OpenAI API key is **never a Terraform resource** — `statements.tf` only grants the processor Lambda's role `ssm:GetParameter` on a parameter path it doesn't itself create. You put the value there yourself via `aws ssm put-parameter` (plain `String`, not `SecureString` — see the KMS gotcha below) before the first `terraform apply` that references it. This keeps the key out of `terraform.tfstate` entirely.
- **State is local** (`infra/terraform.tfstate`, gitignored) — deliberately, to keep first-time setup simple. Never delete this file; it's the only record of what's been created. See `infra/README.md` for how to migrate to an S3 backend later if needed.
- Applied via a dedicated least-privilege IAM user `terraform-personal-site` (NOT the CI/CD `githubActions` user, which is scoped only to S3 sync + CloudFront invalidation). Its policy is scoped to `emrecancioglu-personal-site-*` resources — see `infra/README.md` for the full, current JSON (kept up to date there — check it before assuming a Terraform change needs a *new* AWS resource type's permissions; it may already need updating instead). When adding new resource types, `terraform apply` fails with `AccessDenied` until the policy is extended — extend it, don't assume the Terraform code is wrong.
- Everything AWS-side is designed to stay in the free tier. **Never introduce a billable AWS resource** (customer-managed KMS keys, provisioned capacity, NAT gateways, etc.) without asking first and getting explicit confirmation — this has been rejected before even at ~$1/month. (AWS Budgets itself is free for the first 2 budgets/account, which is why `budget.tf` was fine to add outright.) The one deliberate exception is the OpenAI API (Faz 4, `statements.tf`/`processor.ts`) — usage-based, billed separately by OpenAI, explicitly approved for that specific purpose. Don't treat that approval as blanket permission for other paid APIs.
- To apply: `cd apps/api && npm run build && cd ../../infra && terraform plan` (review!) `&& terraform apply`.

## Deployment

- **Frontend** (`apps/web`): auto-deploys via `.github/workflows/deploy-web.yml` on push to `main` — builds and syncs `apps/web/dist` to S3, then invalidates CloudFront (distribution ID is a GitHub Actions **Variable**, not a Secret — `vars.AWS_CLOUDFRONT_DISTRIBUTION_ID`). Uses `npm install`, not `npm ci` (see comment in the workflow file for why). Checkout has `lfs: true` (the hero video is stored in Git LFS). Note: this workflow does not run `npm run test` before deploying — only builds.
- **Backend** (`apps/api` + `infra`): manual only, via `terraform apply` from a local machine with the `terraform-personal-site` credentials. `.github/workflows/ci-api.yml` runs typecheck/test/build (no deploy, no AWS credentials) on PRs/pushes touching `apps/api/**` as a safety net, but nothing in CI touches AWS infra or the Lambda function itself.
- Only commit/push when explicitly asked — the repo owner does his own commits.

## Known gotchas (don't re-debug these)

- **AWS account quirk (761018862186, eu-north-1)**: the default Lambda-managed KMS key (`alias/aws/lambda`) fails to decrypt environment variables for a Lambda's execution role even with correct IAM + key-policy grants (confirmed reproducible, root cause never fully identified). This also rules out SSM **SecureString** parameters for the same reason (same KMS decrypt path). Workaround pattern for this whole codebase: **no Lambda environment variables, ever** — hardcode non-secret config as build-time constants, and for actual secrets (the OpenAI key), use a plain SSM `String` parameter (not SecureString) read at invocation time via `@aws-sdk/client-ssm`. Don't reintroduce `environment { variables = {...} }` on any `aws_lambda_function` resource, and don't spend time re-diagnosing this — it's a known dead end.
- **TEFAS auto-scraping is off the table.** Investigated thoroughly: no official/free API exists (SPK's public `ws.spk.gov.tr` service doesn't cover fund NAV data; Takasbank's feed is B2B-licensed only), and tefas.gov.tr runs a real bot-defense product (F5-style JS challenge), confirmed blocking requests from two different network origins. Portfolio data entry is intentionally manual, via the admin panel's "fon ekle" / "fiyat ekle" forms. Don't re-propose scraping tefas.gov.tr directly.
- The repo owner was doing Terraform/AWS CLI for the first time this project — if guiding through infra changes, give exact copy-paste commands one step at a time and confirm output before proceeding, don't assume CLI/terminal familiarity.
